# MatchFinder — Technical Debt Register

Living document. Consult before planning any feature (per AGENTS.md workflow step 2). Update whenever debt is added or resolved — mark items resolved with a date rather than deleting them, so history stays auditable.

**Last full audit:** 2026-07-20 (pre-launch product/security audit; see "Shipped: audit remediation Stages A–D" in Resolved for the fix-out).

## 1. Known permissive/duplicated policies (documented as-is, not narrowed)

- [ ] `push_delivery_log`'s "Service role can insert/update delivery logs" policies have `qual: true` / `with_check: true` — any authenticated session can write or alter a delivery-log row for **any** user, not just their own. The naming is historical; this app has no real service-role client (see AGENTS.md), so in practice these are just permissive-for-everyone. Found while backfilling this table's migration (`supabase/migration-schema-drift-backfill.sql`) — recorded as-is since narrowing it is a behavior change, not documentation, and was out of that migration's scope. Low impact (delivery-log rows aren't used for anything security-sensitive today) but should be narrowed to `auth.uid() = user_id` when this table is next touched.
- [ ] `join_requests` has two near-duplicate INSERT policies (`"Create join requests"` and `"Players can insert own join requests"`, both originally `with_check (auth.uid() = player_id)`) and two near-duplicate SELECT policies (`"View join requests"` and `"Organizers can view join requests for their matches"`). Found while adding block-check enforcement (both INSERT policies had to be updated identically, since permissive Postgres RLS policies are OR'd — narrowing only one would have left the other as an unrestricted bypass). Not consolidated; a future touch to this table should merge each pair into one policy.

## Resolved

- [x] (2026-07-20) **Shipped: pre-launch audit remediation, Stages A–D.** Full punch-list from an independent product/security audit (published as an artifact), executed as four checkpointed migrations, each dry-run tested against production inside a rolled-back transaction before being applied for real:
  - **Stage A** (code-only): `/api/cron/*` routes now require a `CRON_SECRET` bearer token (previously callable by anyone); `src/lib/email.ts` now HTML-escapes title/message (closed an injection vector) and no longer double-prepends `https://` when `NEXT_PUBLIC_SITE_URL` already has a protocol; the transactional email template now uses the product's actual brand colors (`#16a34a`/`#0a0e14`) instead of an unrelated indigo/purple gradient; `src/app/api/matches/route.ts` search now strips `,()` before interpolating into the PostgREST filter string, matching the pattern already used elsewhere; Google Fonts `@import` removed from `globals.css` (fonts were double-loaded — `next/font` already covers the same weights).
  - **Stage B** (security/moderation schema — `supabase/migration-security-hardening.sql`): in-memory rate limiting (ineffective on serverless — reset per cold start, not shared across instances) replaced with a durable `check_rate_limit(scope)` RPC backed by `rate_limit_buckets` + a `rate_limit_policies` lookup table (limits are server-side, not caller-supplied — see the corrective fix below); new `moderator` role tier (`is_staff()`, `requireStaff()`) so a bus-factor-of-one on moderation is no longer structural — role changes still require the true admin, status changes (suspend/ban) now work for either; new trigger-based `admin_audit_log` — entries are derived from the actual committed change by a `SECURITY DEFINER` trigger, not an app-level call some route could forget to make, so no client session (including a legitimate staff member's own browser) can insert, alter, or delete an entry.
    - *Corrective fix, found via `get_advisors` after applying*: `check_rate_limit` originally took a caller-supplied key **and** caller-supplied limit — any authenticated caller could hit the RPC directly via PostgREST and either target another user's bucket key or pass an inflated limit to defeat their own. Fixed: key is now always derived from `auth.uid()`, limits are always looked up from `rate_limit_policies`, never caller-supplied.
  - **Stage C** (block/mute — `supabase/migration-blocking-and-chat-rls-fix.sql`): new `blocked_users` table; blocking is enforced at the database layer (not just the app) in three places — `get_or_create_dm` refuses to create a DM between a blocked pair, the `messages` INSERT policy refuses a message into an existing DM where either party has blocked the other, and both `join_requests` INSERT policies refuse a request between a blocked organizer/player pair.
    - **Critical pre-existing bug found and fixed in the same pass**: every RLS policy touching `conversations`, `conversation_participants`, or `messages` recursed infinitely under real authenticated access (`conversation_participants`' own SELECT policy self-references the table, and `conversations`/`messages`' policies also reference `conversation_participants` — nesting a table's self-referential RLS inside another table's RLS is a genuine Postgres infinite-recursion trap). Confirmed via testing: a bare `select * from conversation_participants` recursed under `SET ROLE authenticated`. `messages` had zero rows in production — as far as can be determined, **chat had never successfully sent a message under real RLS**, invisible because every prior manual test in this project's history ran via the `postgres` role (bypasses RLS) rather than `authenticated`. Fixed with a `SECURITY DEFINER` helper (`is_conversation_participant`), the same pattern `is_admin()`/`is_staff()` already use to break equivalent cycles.
    - *Corrective fix, found via `get_advisors` after applying*: `get_or_create_dm(creator, target)` never verified `auth.uid() = creator` — any authenticated caller could pass an arbitrary `creator` id and insert themselves into a DM with an unwilling victim, bypassing consent (though not the block check itself, which still evaluates the real ids). Fixed; the app route already always passed `creator: user.id`, so no legitimate behavior changed.
  - **Stage D** (documentation-only — `supabase/migration-schema-drift-backfill.sql`): backfilled migration files for the 8 tables and 4 `matches` columns tracked in the old "Schema drift" section (`match_templates`, `match_photos`, `match_availability`, `match_checkins`, `activity_feed`, `message_reactions`, `push_subscriptions`, `push_delivery_log`, plus `matches.position_needed`/`price_per_person`/`motm_player_id`/`fair_play_player_id`), captured directly from the live database via `list_tables` introspection. Every statement is `IF NOT EXISTS`/guarded — a no-op against production, applied purely so the schema is reconstructable from the repo. The database can now be rebuilt from `supabase/` alone.
  - Deliberately **not** in this pass: localization and a "regulars/crew" feature (both top-ROI per the audit, both standalone product builds — see "Deferred fast-follows" above); Stage E (automated tests) followed as its own effort — see below.
- [x] (2026-07-20) **Shipped: Stage E test infrastructure.** 39 Vitest tests in `tests/**` run real integration tests against production Postgres via a direct pooler connection (`SUPABASE_TEST_DB_URL` — session pooler, local `.env.test` gitignored, and a matching GitHub Actions secret for CI). Neither Supabase branches nor a local Docker-based stack were available in this environment (branch creation needs a `confirm_cost` step not exposed here; no Docker installed), so tests run directly against production instead — real schema-level isolation wasn't achievable anyway, since the functions under test (`accept_join_request`, `prevent_self_role_escalation`, etc.) are hard-coded to reference `public.profiles`/`public.matches` by name, so a parallel test schema wouldn't actually be exercised by them. Isolation instead comes from every test creating its own throwaway `auth.users` row and running inside `BEGIN...ROLLBACK` (`tests/helpers/db.ts`, `tests/helpers/fixtures.ts`) — fixture setup and assertions are undone atomically together, so nothing is ever visible outside the test's own transaction and a hard crash mid-test can't leave orphaned rows. Tests use `SET ROLE authenticated` + `request.jwt.claims` so RLS is actually enforced — the pool's own default role bypasses RLS entirely, the same trap that hid the chat recursion bug in Stage C.
  - Coverage: `join_requests` RLS (create/withdraw), `accept_join_request` capacity/waitlist atomicity, `prevent_self_role_escalation` for role-only/status-only/combined updates across user/moderator/admin actors, `check_rate_limit` (cap enforcement, per-user bucket isolation, fail-closed on unknown scope/no auth, anon rejection), the chat RLS recursion regression guard, and block enforcement (DM creation, message send, join requests, `blocked_users` RLS scoping).
  - `.github/workflows/ci.yml`: typecheck + build + tests now block PRs; lint runs but is `continue-on-error` (see "Code structure" above — 42 pre-existing errors predate this workflow).
  - **A second critical, actively-exploitable vulnerability was found and fixed while writing these tests**: neither `accept_join_request` nor `remove_accepted_player` verified `auth.uid() = p_organizer_id`, and both were callable by `anon` (fully unauthenticated). Confirmed exploitable before fixing: an anonymous caller who knew a `join_request_id` and the real organizer's user id could accept any pending join request, or kick any accepted player, for a match they don't organize — no account required. Same class of bug as `get_or_create_dm` (Stage C), same fix: verify caller identity, revoke `anon` execute. Both fixed and covered by dedicated `SECURITY REGRESSION` tests in `tests/join-lifecycle.test.ts`.
